# project-docs — Prompt Instructions

## General Rules

1. Always place output files in `docs/<project-name>/`
2. Create the directory if it does not exist
3. Derive `<project-name>` from the current change name, branch name, or ask the user
4. Use the templates below — fill every section, mark sections as "N/A" if not applicable
5. Commit docs to git after generating: `git add docs/ && git commit -m "docs: update <project> documentation"`

---

## Document: design.md

**When:** After brainstorming / design approval

**Source:** Collect from the current session's brainstorming output, design discussions, and any market analysis artifacts.

**Steps:**
1. Identify the project name from context
2. Gather all design decisions made during brainstorming
3. Write `docs/<project>/design.md` using the template in `templates/design.md`
4. Include:
   - Problem statement (from market analysis if available)
   - Proposed solution with architecture overview
   - Key design decisions and **why** (trade-offs considered)
   - Data model / API surface (if applicable)
   - Non-functional requirements (performance, security, scalability)
   - Open questions or deferred decisions
5. Commit to git

---

## Document: plan.md

**When:** After implementation plan is approved

**Source:** Collect from the Superpowers planning phase output.

**Steps:**
1. Read the implementation plan generated by Superpowers `writing-plans` skill
2. Write `docs/<project>/plan.md` using the template in `templates/plan.md`
3. Include:
   - Summary of scope
   - Task list with estimated time (from Superpowers tasks)
   - Dependencies between tasks
   - Acceptance criteria per task
   - Risk / blockers
4. Commit to git

---

## Document: changelog.md

**When:** After implementation finishes (all tasks done)

**Source:** Git history + test results + review notes.

**Steps:**
1. Gather commit history for the project branch: `git log --oneline <branch>`
2. Gather test results summary
3. Write `docs/<project>/changelog.md` using the template in `templates/changelog.md`
4. Include:
   - What was built (feature summary)
   - Files changed (grouped by component)
   - Tests added / modified
   - Breaking changes (if any)
   - Known issues / tech debt
5. Commit to git

---

## Command: sync

**When:** On demand — regenerate all docs

**Steps:**
1. Check which documents exist in `docs/<project>/`
2. Regenerate each document from current context
3. Show diff summary of what changed
4. Commit to git
